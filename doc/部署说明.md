# 学生托管系统部署说明

本文档适用于当前 Node.js + SQLite 版本。推荐在一台 Linux 服务器上使用 Docker Compose 部署；填写域名后，脚本会通过 Caddy 自动申请证书并强制 HTTPS。

## 1. 部署结构

- 应用容器：Node.js 22，监听容器内 `3000` 端口。
- 数据目录：宿主机项目目录下的 `data/`，容器重建不会删除。
- 配置文件：`.env.deploy`，包含管理员初始凭据，不提交 Git。
- 数据库：`data/stumng.sqlite`，同时可能存在 SQLite 的 `-wal`、`-shm` 文件。

生产环境最低建议：1 核 CPU、1GB 内存、10GB 磁盘。服务器应安装 Docker Engine、Docker Compose 插件、OpenSSL 和 curl。推荐使用加入 Docker 用户组的普通部署用户；若必须以 root 运行，脚本会把应用容器固定映射到非 root 的 `10001:10001`，并同步数据目录所有权。

## 2. 一键部署

将代码上传或克隆到服务器后执行：

```bash
chmod +x scripts/deploy.sh
./scripts/deploy.sh deploy
```

首次执行会：

1. 创建权限为仅当前用户可读的 `.env.deploy`。
2. 自动生成 36 位十六进制管理员密码并仅写入 `.env.deploy`，不输出到终端日志。
3. 校验弱密码、用户身份和数据目录权限。
4. 构建镜像、启动容器并挂载数据目录。

请通过受控方式查看 `.env.deploy` 中的 `ADMIN_PASSWORD` 并妥善保存。未配置域名时，应用只监听服务器本机：

```text
http://127.0.0.1:3000
```

该地址不能从公网直接访问，可通过 SSH 端口转发临时测试。正式部署应提前把域名解析到服务器，再首次部署：

```bash
DOMAIN=care.example.com ADMIN_ACCOUNT=admin ADMIN_NAME=系统管理员 ./scripts/deploy.sh deploy
```

脚本会启动 Caddy 并自动开放 80/443、申请证书，正式访问地址为 `https://care.example.com`。也可以复制 `.env.deploy.example` 为 `.env.deploy` 后手动修改，但脚本会拒绝示例密码、少于 12 位的密码、root 容器用户以及公开绑定的 HTTP 端口。

## 3. 常用运维命令

```bash
# 查看状态
./scripts/deploy.sh status

# 查看实时日志
./scripts/deploy.sh logs

# 在线备份数据库
./scripts/deploy.sh backup

# 更新代码后：先备份，再重新构建
./scripts/deploy.sh update

# 暂停服务
./scripts/deploy.sh stop

# 移除容器但保留 data/ 数据
./scripts/deploy.sh down
```

备份文件保存在 `HOST_DATA_DIR/backups/`，默认保留 30 天；可通过 `BACKUP_RETENTION_DAYS` 调整。备份目录不会提交 Git，建议再同步到另一台服务器或对象存储，避免服务器磁盘损坏时备份一并丢失。Docker 应用与代理日志均限制为最多 3 个、每个 10MB。

## 4. HTTPS 与反向代理

一键部署默认将应用端口绑定到 `127.0.0.1`。设置 `DOMAIN` 后自动启用 Caddy，将 HTTP 重定向到 HTTPS，并写入 HSTS。防火墙只需开放 SSH、80 和 443。

如果已有外部 Nginx 或云厂商网关，可以不设置 `DOMAIN`，由外部代理访问本机的 3000 端口。Nginx 的 HTTPS 站点核心配置如下，证书路径由 Certbot 或云厂商管理：

```nginx
server {
    listen 443 ssl http2;
    server_name care.example.com;

    ssl_certificate /etc/letsencrypt/live/care.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/care.example.com/privkey.pem;

    client_max_body_size 64k;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Docker 部署默认设置 `TRUST_PROXY=true`，登录限流会读取可信反向代理提供的首个 `X-Forwarded-For`。由于应用端口只绑定本机，公网用户无法绕过代理直接伪造该请求头。

## 5. 配置说明

| 变量 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `APP_PORT` | 否 | `3000` | 宿主机对外端口 |
| `APP_BIND` | 否 | `127.0.0.1` | HTTP 监听地址，安全校验要求保持本机地址 |
| `DOMAIN` | 否 | 空 | 正式域名；填写后自动启用 Caddy HTTPS |
| `PORT` | 否 | `3000` | 容器内监听端口，Compose 部署不要修改 |
| `HOST` | 否 | 镜像内为 `0.0.0.0` | 容器内部监听地址；宿主机仍由 `APP_BIND` 限制 |
| `DATA_DIR` | 否 | `/app/data` | 容器内数据目录，Compose 部署不要修改 |
| `HOST_DATA_DIR` | 否 | `./data` | 宿主机数据目录；建议使用本机磁盘，不要使用共享网络文件系统 |
| `TRUST_PROXY` | 否 | `true` | 信任部署内的反向代理以获取真实客户端地址 |
| `BACKUP_RETENTION_DAYS` | 否 | `30` | 本机数据库备份保留天数 |
| `APP_IMAGE` | 否 | `stumng:local` | 应用镜像名称，用于更新失败回滚 |
| `ADMIN_ACCOUNT` | 是 | `admin` | 初始化或维护的管理员账号 |
| `ADMIN_PASSWORD` | 是 | 自动生成 | 管理员初始密码；改成一个新的值并执行 deploy/update 可主动轮换 |
| `ADMIN_NAME` | 否 | `系统管理员` | 管理员显示名称 |
| `HOST_UID` / `HOST_GID` | 否 | 当前普通用户；root 时为 `10001` | 容器写入宿主机数据目录时使用的非 root 身份 |

脚本会强制 `.env.deploy` 和数据库文件为 `600`，数据及备份目录为 `700`。手工检查命令：

```bash
chmod 600 .env.deploy
chmod 700 data data/backups
```

环境管理员由稳定标记维护：修改 `ADMIN_ACCOUNT` 后会迁移同一个管理员账号，旧账号不会保留；如果新账号已被其他用户占用，服务会拒绝启动。管理员在网页修改密码后，普通重启不会恢复旧环境密码；若需要通过环境配置轮换，必须填写一个新的 `ADMIN_PASSWORD` 并执行 `deploy` 或 `update`。只修改文件再执行 `docker restart` 不会重新加载配置。

密码通过 HTTPS 传输，数据库使用随机盐和 PBKDF2-SHA256 600,000 次迭代保存，旧版 100,000 次哈希会在用户成功登录后自动升级。数据库中不保存可解密的明文密码。

## 6. 数据备份与恢复

运行中的数据库应使用脚本的在线备份命令，不要只复制主数据库文件；SQLite WAL 模式下，未合并的数据可能仍在 `-wal` 文件中。

默认数据目录的恢复步骤如下；自定义 `HOST_DATA_DIR` 时必须替换为实际路径：

```bash
./scripts/deploy.sh stop
cp data/stumng.sqlite data/stumng.sqlite.before-restore
cp data/backups/stumng-YYYYMMDD-HHMMSS.sqlite data/stumng.sqlite
rm -f data/stumng.sqlite-wal data/stumng.sqlite-shm
./scripts/deploy.sh deploy
```

恢复前必须确认备份文件来源和日期，并保留当前数据库副本。

## 7. 更新与回滚

更新代码后执行：

```bash
git pull --ff-only
./scripts/deploy.sh update
```

脚本会先创建在线数据库备份，在旧容器继续运行时构建新镜像，再替换容器并检查健康状态。新版本未通过检查时会自动恢复上一镜像。若自动回滚也失败：

1. 记录应用日志。
2. 确认当前 Git 提交和 `.env.deploy` 是否兼容。
3. 重新执行 `./scripts/deploy.sh deploy`。
4. 只有数据库结构或数据确实不兼容时才恢复备份。

升级到支持教师/家长身份切换的版本时，首次启动会自动为旧账号补齐身份列表，并回填历史任务、出勤记录的操作身份。该迁移可重复执行；仍建议先通过 `./scripts/deploy.sh backup` 备份，并在升级后抽查历史任务的创建身份和账号切换功能。

## 8. 安全检查清单

- 必须使用随机强管理员密码，并限制 `.env.deploy` 和 `data/` 的读取权限。
- 公网部署必须启用 HTTPS，防火墙只开放 SSH、HTTP 和 HTTPS。
- 不要把 `APP_BIND` 改为 `0.0.0.0`；脚本会拒绝公开明文 HTTP。
- 不要运行 `npm run seed:test`，避免生产环境出现测试账号。
- 定期执行恢复演练，确认备份确实可用。
- 定期安装服务器与 Docker 安全更新。
- SQLite 适合当前单机规模；不要让多个应用实例同时写入同一份数据库文件。

## 9. 不使用 Docker 的部署方式

服务器需安装 Node.js 20 或更高版本以及构建原生依赖所需工具：

```bash
npm ci --omit=dev
mkdir -p data
ADMIN_ACCOUNT=admin ADMIN_PASSWORD='请替换为强密码' NODE_ENV=production npm start
```

长期运行时应使用 systemd 或其他进程管理器，并设置自动重启。非 Docker 部署同样需要配置 HTTPS、限制文件权限并执行 SQLite 在线备份。
