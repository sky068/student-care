# 学生托管系统部署说明

本文档适用于当前 Node.js + SQLite 版本。推荐在一台 Linux 服务器上使用 Docker Compose 部署，并通过 Nginx 或云厂商网关提供 HTTPS。

## 1. 部署结构

- 应用容器：Node.js 22，监听容器内 `3000` 端口。
- 数据目录：宿主机项目目录下的 `data/`，容器重建不会删除。
- 配置文件：`.env.deploy`，包含管理员初始凭据，不提交 Git。
- 数据库：`data/stumng.sqlite`，同时可能存在 SQLite 的 `-wal`、`-shm` 文件。

生产环境最低建议：1 核 CPU、1GB 内存、10GB 磁盘。服务器应安装 Docker Engine 与 Docker Compose 插件。

## 2. 一键部署

将代码上传或克隆到服务器后执行：

```bash
chmod +x scripts/deploy.sh
./scripts/deploy.sh deploy
```

首次执行会：

1. 创建权限为仅当前用户可读的 `.env.deploy`。
2. 自动生成 36 位十六进制管理员密码并在终端显示一次。
3. 构建镜像、启动容器并挂载 `data/`。

请立即保存首次显示的管理员密码。默认访问地址为：

```text
http://服务器IP:3000
```

如需修改端口、管理员账号或管理员名称，部署前设置环境变量：

```bash
APP_PORT=8080 ADMIN_ACCOUNT=admin ADMIN_NAME=系统管理员 ./scripts/deploy.sh deploy
```

也可以复制 `.env.deploy.example` 为 `.env.deploy` 后手动修改。不得继续使用示例密码。

## 3. 常用运维命令

```bash
# 查看状态
./scripts/deploy.sh status

# 查看实时日志
./scripts/deploy.sh logs

# 在线备份数据库
./scripts/deploy.sh backup

# 更新代码后：先备份，再重新构建
./scripts/deploy.sh update

# 暂停服务
./scripts/deploy.sh stop

# 移除容器但保留 data/ 数据
./scripts/deploy.sh down
```

备份文件保存在 `data/backups/`。备份目录不会提交 Git，建议再同步到另一台服务器或对象存储，避免服务器磁盘损坏时备份一并丢失。

## 4. HTTPS 与 Nginx

生产环境不建议直接暴露 3000 端口。可以只允许本机或内网访问应用端口，并使用 Nginx 反向代理：

```nginx
server {
    listen 80;
    server_name care.example.com;

    client_max_body_size 64k;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

配置域名后使用 Certbot 或云厂商证书服务启用 HTTPS。家长和教师的密码、任务与学生信息不应通过明文 HTTP 在公网传输。

注意：当前服务的登录限流按直接连接地址计算。容器部署时直接连接地址通常是容器网关；若后续需要按真实客户端 IP 精确限流，应先在服务端增加可信代理配置，不能直接信任任意请求提供的 `X-Forwarded-For`。

## 5. 配置说明

| 变量 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `APP_PORT` | 否 | `3000` | 宿主机对外端口 |
| `PORT` | 否 | `3000` | 容器内监听端口，Compose 部署不要修改 |
| `DATA_DIR` | 否 | `/app/data` | 容器内数据目录，Compose 部署不要修改 |
| `HOST_DATA_DIR` | 否 | `./data` | 宿主机数据目录；建议使用本机磁盘，不要使用共享网络文件系统 |
| `ADMIN_ACCOUNT` | 是 | `admin` | 初始化或维护的管理员账号 |
| `ADMIN_PASSWORD` | 是 | 自动生成 | 管理员密码，修改后重启会同步更新并使旧会话失效 |
| `ADMIN_NAME` | 否 | `系统管理员` | 管理员显示名称 |
| `HOST_UID` / `HOST_GID` | 否 | 当前用户 | 容器写入宿主机数据目录时使用的身份 |

`.env.deploy` 权限建议保持为 `600`：

```bash
chmod 600 .env.deploy
```

## 6. 数据备份与恢复

运行中的数据库应使用脚本的在线备份命令，不要只复制主数据库文件；SQLite WAL 模式下，未合并的数据可能仍在 `-wal` 文件中。

恢复步骤：

```bash
./scripts/deploy.sh stop
cp data/stumng.sqlite data/stumng.sqlite.before-restore
cp data/backups/stumng-YYYYMMDD-HHMMSS.sqlite data/stumng.sqlite
rm -f data/stumng.sqlite-wal data/stumng.sqlite-shm
./scripts/deploy.sh deploy
```

恢复前必须确认备份文件来源和日期，并保留当前数据库副本。

## 7. 更新与回滚

更新代码后执行：

```bash
git pull --ff-only
./scripts/deploy.sh update
```

脚本会先创建在线数据库备份，再重新构建容器。若更新异常：

1. 记录应用日志。
2. 切回上一个确认可用的 Git 提交。
3. 重新执行 `./scripts/deploy.sh deploy`。
4. 只有数据库结构或数据确实不兼容时才恢复备份。

升级到支持教师/家长身份切换的版本时，首次启动会自动为旧账号补齐身份列表，并回填历史任务、出勤记录的操作身份。该迁移可重复执行；仍建议先通过 `./scripts/deploy.sh backup` 备份，并在升级后抽查历史任务的创建身份和账号切换功能。

## 8. 安全检查清单

- 必须使用随机强管理员密码，并限制 `.env.deploy` 和 `data/` 的读取权限。
- 公网部署必须启用 HTTPS，防火墙只开放 SSH、HTTP 和 HTTPS。
- 不要运行 `npm run seed:test`，避免生产环境出现测试账号。
- 定期执行恢复演练，确认备份确实可用。
- 定期安装服务器与 Docker 安全更新。
- SQLite 适合当前单机规模；不要让多个应用实例同时写入同一份数据库文件。

## 9. 不使用 Docker 的部署方式

服务器需安装 Node.js 20 或更高版本以及构建原生依赖所需工具：

```bash
npm ci --omit=dev
mkdir -p data
ADMIN_ACCOUNT=admin ADMIN_PASSWORD='请替换为强密码' NODE_ENV=production npm start
```

长期运行时应使用 systemd 或其他进程管理器，并设置自动重启。非 Docker 部署同样需要配置 HTTPS、限制文件权限并执行 SQLite 在线备份。
